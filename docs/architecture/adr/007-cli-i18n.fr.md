# ADR-007 — Internationalisation (i18n) du CLI openhub

## Statut

Accepté

## Contexte

Tous les messages utilisateur du CLI (`oh`) étaient codés en dur en français dans les
scripts shell. Cette situation posait plusieurs problèmes :

- Les équipes anglophones ne pouvaient pas utiliser le CLI dans leur langue.
- Les messages mélangés (commentaires en FR, code en EN) rendaient la maintenance difficile.
- L'ADR-005 avait introduit l'adaptation linguistique des *agents* déployés, mais le CLI
  lui-même restait monolingue.

La question est : comment rendre tous les messages du CLI traduisibles sans alourdir les
scripts et sans rompre la compatibilité avec bash 3.2 (macOS).

## Décision

**Option A retenue** : table de chaînes centralisée en `scripts/lib/i18n.sh` avec une
fonction `t()` basée sur des `case` statements (compatible bash 3.2).

### Priorité de résolution de langue

```
projet (Langue field) > global (hub.json .cli.language) > défaut (en)
```

- Le champ `Langue` d'un projet dans `projects.md` prend la priorité absolue.
- La clé `cli.language` dans `config/hub.json` configure la langue globale du CLI.
- La valeur de repli codée dans `t()` est `en` (anglais).
- Le fichier `config/hub.json` livré dans le repo contient `"language": "fr"` pour
  que les suites de tests (rédigées en français) passent sans configuration supplémentaire.

### Architecture

```
scripts/lib/i18n.sh        — table EN/FR + fonctions t() et t_en()
scripts/common.sh          — get_hub_language(), resolve_oc_lang(), resolve_agent_lang()
                             + auto-appel de resolve_oc_lang au chargement
config/hub.json            — .cli.language : "fr" | "en"
projects/projects.md       — champ Langue : par projet (optionnel)
```

### Commandes CLI

```bash
oh config set language fr   # définir la langue globale
oh config get language       # lire la langue globale
```

## Implémentation

### 1. `scripts/lib/i18n.sh`

Contient deux fonctions :

- `t_en(key)` : renvoie la chaîne anglaise pour `key`.
- `t(key)` : renvoie la chaîne dans `OH_LANG` ; si `OH_LANG=fr` → chaîne FR, sinon
  délègue à `t_en()`. Retombe sur la clé elle-même si non trouvée.

Compatibilité bash 3.2 : pas de `declare -A`. Utilise des `case` statements imbriqués.

### 2. `scripts/common.sh`

Trois nouvelles fonctions :

- `get_hub_language()` : lit `.cli.language` dans `hub.json` via `jq`.
- `resolve_oc_lang([project_id])` : applique la priorité ci-dessus et exporte `OH_LANG`.
- `resolve_agent_lang(raw_lang)` : si `raw_lang` non vide, le renvoie tel quel
  (valeur du champ `Langue`) ; sinon, mappe `OH_LANG` vers le nom lisible
  (`fr` → `"français"`, `en` → `"english"`) pour l'injection dans les agents.

`resolve_oc_lang` est appelé automatiquement en fin de fichier : toute source de
`common.sh` dispose immédiatement de `OH_LANG`.

### 3. Wiring dans les scripts

Tous les scripts `cmd-*.sh` et `oc.sh` ont été mis à jour pour remplacer les chaînes
codées en dur par des appels `t(key)`.

Les adapters (`opencode.adapter.sh`) et `cmd-deploy.sh`
utilisent désormais `resolve_agent_lang` pour propager la langue dans les agents déployés,
en tenant compte du fallback vers `OH_LANG` quand aucun champ `Langue` n'est configuré.

## Options rejetées

### Option B — Fichiers de messages séparés (`.po` / JSON)

Utiliser des fichiers de traduction standard (gettext `.po`, ou JSON) chargés à l'exécution.

**Rejetée** : nécessite `gettext` ou `jq` pour chaque lookup de message — dépendances
non garanties sur tous les environnements cibles. La complexité ajoutée n'est pas
justifiée pour deux langues.

### Option C — Scripts dupliqués par langue

Maintenir une copie `cmd-help.fr.sh`, `cmd-help.en.sh`, etc.

**Rejetée** : maintenance ×2, divergences inévitables.

## Conséquences

- Tous les messages utilisateur du CLI sont traduisibles via `t()`.
- Ajout de 150+ clés EN/FR dans `i18n.sh` (source of truth unique).
- La langue par défaut effective est `fr` (via `hub.json`) pour rester compatible
  avec les tests existants ; le défaut *code* reste `en` (fallback `t()`).
- Les scripts shell restent bash 3.2 compatibles.
- `oh config set language` / `oh config get language` permettent de changer la langue
  sans modifier `hub.json` manuellement.
- Les agents déployés bénéficient du fallback `OH_LANG` → nom de langue lisible
  pour l'instruction d'adaptation linguistique (ADR-005 étendu).
