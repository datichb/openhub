# ADR-007 — CLI internationalisation (i18n) for opencode-hub

## Status

Accepted

## Context

All CLI (`oc`) user-facing messages were hard-coded in French inside shell scripts.
This caused several issues:

- English-speaking teams could not use the CLI in their language.
- Mixed-language scripts (French comments, English code) were harder to maintain.
- ADR-005 introduced language adaptation for *deployed agents*, but the CLI itself
  remained single-language.

The question is: how to make all CLI messages translatable without bloating the scripts
and without breaking bash 3.2 compatibility (macOS ships bash 3.2).

## Decision

**Option A chosen**: centralised string table in `scripts/lib/i18n.sh` with a `t()`
function based on `case` statements (bash 3.2 compatible).

### Language resolution priority

```
project (Langue field) > global (hub.json .cli.language) > default (en)
```

- The `Langue` field of a project in `projects.md` takes highest priority.
- The `cli.language` key in `config/hub.json` sets the global CLI language.
- The built-in fallback in `t()` is `en` (English).
- The `config/hub.json` shipped in the repo contains `"language": "fr"` so the
  test suites (written in French) pass without extra configuration.

### Architecture

```
scripts/lib/i18n.sh        — EN/FR string table + t() and t_en() functions
scripts/common.sh          — get_hub_language(), resolve_oc_lang(), resolve_agent_lang()
                             + auto-call of resolve_oc_lang on source
config/hub.json            — .cli.language: "fr" | "en"
projects/projects.md       — Langue field: per-project override (optional)
```

### CLI commands

```bash
oc config set language fr   # set global language
oc config get language       # read global language
```

## Implementation

### 1. `scripts/lib/i18n.sh`

Two functions:

- `t_en(key)`: returns the English string for `key`.
- `t(key)`: returns the string in `OC_LANG`; if `OC_LANG=fr` → FR string, otherwise
  delegates to `t_en()`. Falls back to the key itself if not found.

Bash 3.2 compatible: no `declare -A`. Uses nested `case` statements.

### 2. `scripts/common.sh`

Three new functions:

- `get_hub_language()`: reads `.cli.language` from `hub.json` via `jq`.
- `resolve_oc_lang([project_id])`: applies the priority chain above and exports `OC_LANG`.
- `resolve_agent_lang(raw_lang)`: if `raw_lang` is non-empty, returns it as-is
  (value of the `Langue` field); otherwise maps `OC_LANG` to a human-readable name
  (`fr` → `"français"`, `en` → `"english"`) for injection into deployed agents.

`resolve_oc_lang` is called automatically at the end of the file: any script that
sources `common.sh` immediately has a correct `OC_LANG`.

### 3. Wiring in scripts

All `cmd-*.sh` scripts and `oc.sh` have been updated to replace hard-coded strings
with `t(key)` calls.

The adapters (`opencode.adapter.sh`) and `cmd-deploy.sh`
now use `resolve_agent_lang` to propagate the language into deployed agents, including
the fallback to `OC_LANG` when no per-project `Langue` field is configured.

## Rejected options

### Option B — Separate message files (`.po` / JSON)

Use standard translation files (gettext `.po`, or JSON) loaded at runtime.

**Rejected**: requires `gettext` or `jq` for every message lookup — not guaranteed
on all target environments. The added complexity is not justified for two languages.

### Option C — Language-duplicated scripts

Maintain `cmd-help.fr.sh`, `cmd-help.en.sh`, etc.

**Rejected**: ×2 maintenance burden, inevitable drift between copies.

## Consequences

- All CLI user-facing messages are translatable via `t()`.
- 150+ EN/FR keys added in `i18n.sh` (single source of truth).
- The effective default language is `fr` (via `hub.json`) for backward compatibility
  with existing tests; the *code* default remains `en` (fallback in `t()`).
- Shell scripts remain bash 3.2 compatible.
- `oc config set language` / `oc config get language` let users change the language
  without editing `hub.json` manually.
- Deployed agents benefit from the `OC_LANG` → human-readable language name fallback
  for the language adaptation instruction (ADR-005 extended).
