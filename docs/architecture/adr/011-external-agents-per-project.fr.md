# ADR-011 — Agents et skills externes par projet : découverte, intégration et substitution

## Statut

Accepté

## Contexte

Le hub opencode-hub déploie ses propres agents (`agents/`) et skills (`skills/`) dans chaque projet cible via `oc deploy`. Cependant, de nombreux projets existants ont déjà des agents définis dans leur `.opencode/agents/` — soit écrits manuellement, soit générés par leur propre outil — sans lien avec le hub.

Deux cas se posaient :

1. **Projets avec agents manuels** : des fichiers `.md` écrits à la main dans `.opencode/agents/`, souvent un `planner`, un `developer` ou un `reviewer` spécifiques au projet. Le deploy hub les **écrasait** silencieusement.

2. **Projets avec leur propre hub d'agents** : certains projets ont une structure d'agents similaire au hub (souvent une organisation `agents/planning/`, `agents/developer/`) avec des agents adaptés à leur domaine métier. Forcer la migration vers le hub central n'est pas toujours souhaitable.

Dans les deux cas, le besoin est le même : **laisser coexister les agents du projet avec les agents du hub**, avec un mécanisme clair pour décider agent par agent si l'un substitue l'autre ou s'y ajoute.

Un troisième enjeu est la **pertinence de la détection** : nommer un agent `planner` dans un projet doit être reconnu comme similaire à l'agent `planner` du hub, mais aussi `plan`, `planning` ou `project-planner`. À l'inverse, `dev-frontend` doit mapper sur `developer-frontend`, pas sur `developer-fullstack`.

## Décision

### 1. Table d'aliases sémantiques (`config/agent-aliases.json`)

Un fichier JSON centralisé organise les correspondances entre noms courants (venant de projets externes) et IDs hub canoniques. Couverture : toutes les familles du hub, variantes anglaises et françaises, synonymes métier, nommages OpenCode natifs (`plan`, `explore`, etc.).

La résolution fonctionne en 3 niveaux :
- **Niveau 1** — match exact de l'ID normalisé avec un ID hub (ex: `planner` → `planner`)
- **Niveau 2** — lookup dans `agent-aliases.json` (ex: `plan` → `planner`, `frontend` → `developer-frontend`)
- **Niveau 3** — strip de préfixes courants (`dev-`, `my-`, `agent-`) puis re-lookup (ex: `my-planner` → `planner`)

### 2. Bibliothèque de découverte (`scripts/lib/agent-discovery.sh`)

Trois fonctions publiques :
- `discover_project_agents(project_path)` — scanne `<projet>/.opencode/agents/*.md`, filtre ceux **sans** le marker `<!-- generated by opencode-hub` (agents non générés par le hub)
- `resolve_agent_similarity(agent_id)` — applique les 3 niveaux de résolution, retourne les IDs hub correspondants ou vide
- `is_hub_generated_agent(file)` — teste la présence du marker hub (5 premières lignes)
- `get_discovery_agent_id(file)` — lit le frontmatter `id:`, normalise, fallback sur le basename

### 3. Persistance dans `projects.md` (champ `External agents`)

Les choix de l'utilisateur sont persistés dans le champ `External agents` du bloc projet :

```markdown
- External agents : .opencode/agents/my-planner.md:substitute:planner|.opencode/agents/custom.md:complement
```

Format par entrée, séparateur `|` :
- `<chemin>:substitute:<hub-id>` — l'agent projet **remplace** l'agent hub `<hub-id>` pour ce projet
- `<chemin>:complement` — l'agent projet **s'ajoute** aux agents hub sans en remplacer aucun

Les chemins sont **relatifs au `project_path`** résolu (ou absolus si commençant par `/`).

### 4. Découverte automatique au deploy (`_cmd_deploy_discover` dans `cmd-deploy.sh`)

Avant la Phase 1 du deploy, si un `PROJECT_ID` est fourni et que le contexte est interactif (TTY + `OC_NON_INTERACTIVE != 1`) :
1. `discover_project_agents()` scanne `.opencode/agents/`
2. Les agents non déjà configurés dans `External agents` sont présentés à l'utilisateur
3. Pour chaque agent : si un match hub est trouvé → `[s]ubstituer / [c]omplément / [i]gnorer` ; sinon → `[c]omplément / [i]gnorer`
4. Les choix sont persistés dans `projects.md` avant de passer à la Phase 1

En mode CI (`OC_NON_INTERACTIVE=1`) : skip silencieux, le deploy continue normalement.

### 5. Sous-commande `oc agent discover PROJECT_ID`

Permet de déclencher la découverte à la demande, sans lancer un deploy complet. Affiche un résumé de la configuration résultante et rappelle la commande deploy à exécuter.

### 6. Application au deploy (`_load_agent_metadata` dans `opencode.adapter.sh`)

Après le scan de `$CANONICAL_AGENTS_DIR`, la fonction applique les entrées `External agents` :
- **Substitutions** : cherche l'index du `hub-id` dans les tableaux `_DEPLOY_FILES_*` et remplace son `_DEPLOY_FILES_AGENT_FILES[i]` par le chemin de l'agent projet
- **Compléments** : ajoute de nouvelles entrées dans les tableaux, en vérifiant l'absence de doublon d'ID

Les agents de substitution sont **déployés tels quels** (`build_agent_content` est appelé normalement — si le frontmatter déclare des `skills:/native_skills:`, ils sont résolus ; sinon, le body est utilisé sans injection supplémentaire).

## Conséquences

### Positives

- Intégration **non-intrusive** : les projets avec agents existants ne sont pas forcés à migrer.
- Découverte **progressive** : les choix sont persistés, le deploy suivant n'interrompt plus l'utilisateur.
- Résolution **pertinente** : `plan`, `planner`, `my-planner` sont tous reconnus comme similaires à `planner` hub.
- **Complète compatibilité descendante** : les projets sans `External agents` dans leur `projects.md` ont un comportement identique à avant.
- Les agents de substitution peuvent avoir leur propre frontmatter (`skills:`, `native_skills:`, `mode:`) qui sera respecté.

### Négatives / compromis

- Le champ `External agents` dans `projects.md` peut devenir long si un projet a beaucoup d'agents personnalisés.
- La table `agent-aliases.json` doit être maintenue manuellement quand de nouveaux agents hub sont ajoutés.
- En cas de substitution d'un agent hub, les skills Bucket A habituellement injectées dans cet agent ne le seront plus (sauf si déclarées dans le frontmatter de l'agent projet). C'est un compromis intentionnel : l'agent projet est utilisé avec **son propre contexte**.
- Les stack skills dynamiques (ADR-008) ne s'appliquent qu'aux agents générés par le hub via `build_agent_content` — les agents de substitution n'en bénéficient pas.

## Alternatives rejetées

**Scan automatique sans prompt** : détecter et intégrer automatiquement les agents non-hub sans demander l'utilisateur. Rejeté car silencieux et risqué — remplacer `planner` sans confirmation peut introduire des comportements inattendus.

**Migration forcée vers le hub** : demander à l'utilisateur de copier ses agents dans `agents/` du hub. Rejeté car intrusif, casse la séparation hub/projet, et empêche les projets d'avoir des agents spécifiques à leur domaine.

**Champ de liste de chemins simple** : stocker uniquement les chemins sans l'action (substitute/complement). Rejeté car ambigu : le hub ne peut pas décider seul de substituer ou compléter.

**Fichier JSON dédié** (`projects/external-agents.local.json`) : stockage dans un fichier séparé plutôt que dans `projects.md`. Rejeté car fragment la configuration projet sur plusieurs fichiers, en rupture avec le modèle existant de `projects.md` comme registre centralisé.

## Relation avec les autres ADRs

- **ADR-001** (Séparation Agent/Skill) : évolué — le mécanisme de deploy peut maintenant utiliser des fichiers agents extérieurs à `agents/`.
- **ADR-008** (Injection dynamique des stack skills) : non affecté — les stack skills continuent de s'appliquer aux agents hub uniquement.
- **ADR-010** (Architecture hybride des skills) : non affecté — les Buckets A et B continuent de fonctionner normalement pour les agents hub.
