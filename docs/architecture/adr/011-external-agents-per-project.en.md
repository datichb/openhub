# ADR-011 вЂ” External Agents per Project: Discovery, Integration and Substitution

## Status

Accepted

## Context

The openhub deploys its own agents (`agents/`) and skills (`skills/`) into each target project via `oh deploy`. However, many existing projects already have agents defined in their `.opencode/agents/` вЂ” either written by hand or generated by their own tooling вЂ” with no connection to the hub.

Two situations arose:

1. **Projects with manual agents**: hand-written `.md` files in `.opencode/agents/`, often a project-specific `planner`, `developer`, or `reviewer`. The hub deploy was **silently overwriting** them.

2. **Projects with their own agent hub**: some projects have an agent structure similar to the hub (often an `agents/planning/`, `agents/developer/` layout) with agents tailored to their business domain. Forcing migration to the central hub is not always desirable.

In both cases the need is the same: **let project agents coexist with hub agents**, with a clear mechanism for deciding agent by agent whether one substitutes the other or is added alongside it.

A third challenge is **detection relevance**: naming an agent `planner` in a project should be recognized as similar to the hub's `planner` agent, but also `plan`, `planning`, or `project-planner`. Conversely, `dev-frontend` should map to `developer-frontend`, not to `developer-fullstack`.

## Decision

### 1. Semantic alias table (`config/agent-aliases.json`)

A centralized JSON file organizes mappings between common names (from external projects) and canonical hub IDs. Coverage: all hub families, English and French variants, business synonyms, native OpenCode agent names (`plan`, `explore`, etc.).

Resolution works in 3 levels:
- **Level 1** вЂ” exact match of the normalized ID with a hub ID (e.g. `planner` в†’ `planner`)
- **Level 2** вЂ” lookup in `agent-aliases.json` (e.g. `plan` в†’ `planner`, `frontend` в†’ `developer-frontend`)
- **Level 3** вЂ” strip common prefixes (`dev-`, `my-`, `agent-`) then re-lookup (e.g. `my-planner` в†’ `planner`)

### 2. Discovery library (`scripts/lib/agent-discovery.sh`)

Four public functions:
- `discover_project_agents(project_path)` вЂ” scans `<project>/.opencode/agents/*.md`, filters those **without** the `<!-- generated by openhub` marker
- `resolve_agent_similarity(agent_id)` вЂ” applies the 3 resolution levels, returns matching hub IDs or empty
- `is_hub_generated_agent(file)` вЂ” tests for the hub marker in the first 5 lines
- `get_discovery_agent_id(file)` вЂ” reads the frontmatter `id:`, normalizes, falls back to basename

### 3. Persistence in `projects.md` (`External agents` field)

User choices are persisted in the `External agents` field of the project block:

```markdown
- External agents : .opencode/agents/my-planner.md:substitute:planner|.opencode/agents/custom.md:complement
```

Entry format, `|` separator:
- `<path>:substitute:<hub-id>` вЂ” the project agent **replaces** hub agent `<hub-id>` for this project
- `<path>:complement` вЂ” the project agent **is added** alongside hub agents without replacing any

Paths are **relative to the resolved `project_path`** (or absolute if starting with `/`).

### 4. Automatic discovery at deploy time (`_cmd_deploy_discover` in `cmd-deploy.sh`)

Before Phase 1 of deploy, if a `PROJECT_ID` is provided and the context is interactive (TTY + `OH_NON_INTERACTIVE != 1`):
1. `discover_project_agents()` scans `.opencode/agents/`
2. Agents not yet configured in `External agents` are presented to the user
3. For each agent: if a hub match is found в†’ `[s]ubstitute / [c]omplement / [i]gnore`; otherwise в†’ `[c]omplement / [i]gnore`
4. Choices are persisted in `projects.md` before proceeding to Phase 1

In CI mode (`OH_NON_INTERACTIVE=1`): silent skip, deploy continues normally.

### 5. `oh agent discover PROJECT_ID` sub-command

Allows triggering discovery on demand without running a full deploy. Displays a summary of the resulting configuration and reminds the user of the deploy command to execute.

### 6. Application at deploy time (`_load_agent_metadata` in `opencode.adapter.sh`)

After scanning `$CANONICAL_AGENTS_DIR`, the function applies `External agents` entries:
- **Substitutions**: finds the index of `hub-id` in the `_DEPLOY_FILES_*` arrays and replaces its `_DEPLOY_FILES_AGENT_FILES[i]` with the project agent path
- **Complements**: appends new entries to the arrays, checking for duplicate IDs

Substitution agents are **deployed as-is** (`build_agent_content` is called normally вЂ” if the frontmatter declares `skills:/native_skills:`, they are resolved; otherwise the body is used without additional injection).

## Consequences

### Positive

- **Non-intrusive integration**: projects with existing agents are not forced to migrate.
- **Progressive discovery**: choices are persisted, subsequent deploys no longer interrupt the user.
- **Relevant resolution**: `plan`, `planner`, `my-planner` are all recognized as similar to hub's `planner`.
- **Full backward compatibility**: projects without `External agents` in their `projects.md` behave identically to before.
- Substitution agents can have their own frontmatter (`skills:`, `native_skills:`, `mode:`) which will be respected.

### Negative / trade-offs

- The `External agents` field in `projects.md` can become long if a project has many custom agents.
- The `agent-aliases.json` table must be manually maintained when new hub agents are added.
- When substituting a hub agent, the Bucket A skills normally injected into that agent will no longer be present (unless declared in the project agent's frontmatter). This is an intentional trade-off: the project agent is used **with its own context**.
- Dynamic stack skills (ADR-008) only apply to hub-generated agents via `build_agent_content` вЂ” substitution agents do not benefit from them.

## Rejected Alternatives

**Automatic scan without prompt**: detect and integrate non-hub agents silently without user confirmation. Rejected as silent and risky вЂ” replacing `planner` without confirmation may introduce unexpected behavior.

**Forced migration to hub**: ask the user to copy their agents into the hub's `agents/`. Rejected as intrusive, breaks the hub/project separation, and prevents projects from having domain-specific agents.

**Simple path list field**: store only paths without the action (substitute/complement). Rejected as ambiguous: the hub cannot decide on its own whether to substitute or complement.

**Dedicated JSON file** (`projects/external-agents.local.json`): storage in a separate file rather than in `projects.md`. Rejected as it fragments project configuration across multiple files, breaking the existing model of `projects.md` as the centralized registry.

## Relationship with Other ADRs

- **ADR-001** (Agent/Skill Separation): evolved вЂ” the deploy mechanism can now use agent files from outside `agents/`.
- **ADR-008** (Dynamic Stack Skills Injection): unaffected вЂ” stack skills continue to apply to hub agents only.
- **ADR-010** (Hybrid Skills Architecture): unaffected вЂ” Buckets A and B continue to work normally for hub agents.
